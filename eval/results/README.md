# Eval Results Guide

This folder stores machine-readable evaluation artifacts generated by `python -m eval.harness`.

Result modes:
- `live_api`: one-shot capture run using real model/retrieval calls
- `replay_benchmark`: deterministic replay of a frozen benchmark artifact
- `offline_fixture`: synthetic smoke validation only

Current branch status: the checked-in timestamped artifacts are still smoke artifacts. The replay benchmark path is implemented, but the repo still needs a frozen benchmark file created from a live capture.

## Tracked Metrics

- `citation_accuracy`: Fraction of extracted claims marked verified by the verifier.
- `completeness`: Fraction of planner sub-questions addressed in the final answer.
- `hallucination_rate`: Fraction of extracted claims not supported by retrieved evidence.
- `uncertainty_reported` (unanswerable only): Whether the answer explicitly signals missing/insufficient evidence.
- `conflict_acknowledged` (conflicting_evidence only): Whether the answer surfaces both sides of a genuine evidential conflict rather than collapsing to a single position.
- `tool_calls`: Approximate end-to-end API call count for efficiency/cost proxy.
- `posthoc_verifier_for_scoring`: Whether a no-verify ablation was scored with a separate verifier pass so claim-level metrics remain comparable.

Metrics that are still missing and should be measured separately before making stronger claims:
- verifier rejection rate on supported paraphrases,
- failure-mode distribution from a frozen replay benchmark,
- measured serial-vs-async latency deltas from frozen timing fields,
- official GAIA validation-set accuracy.

## Experiment Configurations

- `no_plan_no_verify`: Uses original question directly and skips claim verification.
- `plan_no_verify`: Adds decomposition into sub-questions but skips claim verification.
- `plan_verify`: Full pipeline with planning, retrieval/compression, synthesis, and claim-level verification.

## Interpreting Results

- Higher `completeness` means broader answer coverage.
- Lower `hallucination_rate` is better; this is the most important safety metric.
- `citation_accuracy` and `hallucination_rate` are linked but not identical:
  - `citation_accuracy` focuses on supported claims.
  - `hallucination_rate` focuses on unsupported claims.
- For unanswerable tasks, a good model should avoid fabrication and increase `uncertainty_reported`.
- For conflicting_evidence tasks, `conflict_acknowledged` is the primary signal. An agent with low hallucination rate but `conflict_acknowledged=false` has probably picked one side and presented it as consensus — the harder failure to detect.

## File Conventions

- Harness output: `YYYYMMDDTHHMMSSZ.json`.
- Result mode field:
  - `live_api`: real model + retrieval calls
  - `replay_benchmark`: deterministic replay of a frozen artifact
  - `offline_fixture`: deterministic synthetic smoke-run output (no API calls)
- `benchmark_claims_allowed`: whether the artifact is acceptable for benchmark reporting.
- `task_profile`: optional named benchmark profile used for the run.
- Frozen benchmark artifacts live in `eval/replay/`.
- Standard logging schema template: `result_template.json.example`.
- Human summary: generated with `python -m eval.summarize_results --latest`.
- Benchmark-safe summary gate: `python -m eval.summarize_results --latest --mode replay_benchmark --require-benchmarkable`.

## Recommended Workflow

1. Capture once: `python -m eval.harness --profile core_live_28`
2. Freeze it: `python -m eval.freeze_benchmark --run-file eval/results/<run>.json --output eval/replay/core_live_28.json --benchmark-name core_live_28`
3. Zero-cost benchmark rerun: `python -m eval.harness --offline --replay-benchmark eval/replay/core_live_28.json`
4. Summarize latest: `python -m eval.summarize_results --latest --mode replay_benchmark --require-benchmarkable`
5. Synthetic smoke run when needed: `python -m eval.harness --synthetic-smoke`


## Rigor Guardrails

- Do not present `offline_fixture` results as model quality evidence.
- Use `replay_benchmark` for reproducible reporting after a live capture is frozen.
- Distinguish the local 12-task `gaia_l1` subset from the official GAIA validation benchmark.
- Keep synthetic, live, and replay artifacts distinct in interpretation.
